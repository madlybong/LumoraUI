# Plan: Fix npm packaging to stop leaking TS/Vue source diagnostics

## Objective
Ensure `@astrake/lumora-ui` ships compiled artifacts (`dist/*.js` + `dist/*.d.ts`) and no longer exposes raw `.ts`/`.vue` implementation files as runtime/type entrypoints, so downstream strict TypeScript checks are not coupled to library internals.

## Findings from `bug-report.md` and repo
- The bug report is valid: `packages/core/package.json` currently exports only `./src/*.ts` entries.
- `packages/core` currently has no dedicated library bundler config; root `tools/build.ts` runs `vue-tsc` only for core and Vite build only for showcase.
- The reported source-level diagnostics are present in current code (unused imports/params, one missing type-only import).
- Current packaging includes `src` in published files and does not include `dist`.

## Recommended fix strategy
Adopt **Option A (compiled publish)** as the primary fix, and include **targeted source hygiene** from Option B to keep local checks clean.

Rationale:
- Compiled publish is the standard contract for consumer libraries and decouples consumer compiler strictness from internals.
- Source hygiene still improves maintainability and prevents noise in this repo’s own `vue-tsc` checks.

## Implementation plan

### Phase 1 — Add package build outputs for `packages/core`
1. Create a library build config for core (Vite library mode) that emits ESM JS to `packages/core/dist`.
   - Entry: `packages/core/src/index.ts`
   - Externalize peer/runtime deps expected to stay external (`vue`, optionally `shiki`, `clsx`, `tailwind-merge` depending on current bundling policy).
2. Add declaration generation step for core to emit `.d.ts` into `packages/core/dist`.
   - Use `vue-tsc --declaration --emitDeclarationOnly --outDir dist` against `packages/core/tsconfig.json`.
3. Update `packages/core/package.json` scripts:
   - `build`: run declaration build + JS bundle.
   - Keep `check` for strict typecheck only.

### Phase 2 — Switch published entrypoints to `dist`
4. Update `packages/core/package.json` fields:
   - Add `main`, `module`, and `types` pointing to `dist/index.js` and `dist/index.d.ts`.
   - Rewrite `exports` to map each public subpath (`.`, `./components`, `./layout`, `./shell`, `./composables`, `./skins`, `./tailwind`) to compiled JS + types in `dist`.
   - Keep `./style` mapped to `./src/lumora.css`.
5. Update `files` to publish only required artifacts:
   - Include `dist`, `src/lumora.css`, `README.md`, `CHANGELOG.md`.
   - Exclude raw source tree from package payload.

### Phase 3 — Emit declarations for subpath exports
6. Ensure declaration emit includes index files for each exported subpath so `exports["./*"]` has matching `.d.ts` in `dist`.
   - Verify `dist/components/index.d.ts`, `dist/layout/index.d.ts`, etc. are generated.
   - If any path is missing, adjust entry/declaration setup accordingly.

### Phase 4 — Internal source hygiene (Option B subset)
7. Fix known diagnostics that affect this repo’s own strict checks:
   - `packages/core/src/components/LuToggleGroup.vue`: remove unused `Ref` import.
   - `packages/core/src/composables/useSplit.ts`: mark intentionally unused params with `_` prefix or consume them minimally.
   - `packages/core/src/composables/useTheme.ts`: remove unused `watchEffect` import.
   - `packages/core/src/context.ts`: remove unused `SkinMap` import.
   - `packages/core/src/layout/LuDock.vue`: use `import type { VNode }`.
   - `packages/core/src/layout/LuSplit.vue`: remove unused `splitRef` if not required by template logic.
   - `packages/core/src/shell/desktop/LuDesktopRailBar.vue`: remove unused `inject` import.

### Phase 5 — Align root tooling with new package build
8. Update `tools/build.ts` to invoke the new `packages/core` build script (not just `vue-tsc` check), then build showcase.
9. Keep `tools/check.ts` focused on typechecks (core + showcase).

### Phase 6 — Documentation and changelog
10. Update consumer-facing docs that currently reference scanning `node_modules/@astrake/lumora-ui/src` for Tailwind.
   - Preferred future-safe guidance: scan package root or a helper-derived location compatible with compiled publish.
11. Add changelog entry under `Unreleased` describing packaging fix and downstream impact.

## Validation plan
- Run `bun run check` (workspace typecheck).
- Run `bun test`.
- Run `bun run build` and verify `packages/core/dist` contains JS + declarations for all exported subpaths.
- Run `bun pm pack` in `packages/core` and inspect tarball contents:
  - contains `dist/**`, `src/lumora.css`, `README.md`, `CHANGELOG.md`
  - does not expose source `.ts`/`.vue` entrypoints.
- Smoke-check consumer resolution expectations:
  - `types` resolve to `dist/*.d.ts`
  - runtime imports resolve to `dist/*.js`.

## Risks and mitigations
- **Risk:** Subpath export mismatch (JS present, types missing).
  - **Mitigation:** Explicitly verify each `exports` subpath has both artifacts in `dist`.
- **Risk:** Tailwind consumer docs become stale if they still point to `/src`.
  - **Mitigation:** Update docs in same change and preserve helper guidance.
- **Risk:** Bundling policy changes dependency behavior.
  - **Mitigation:** Keep peer deps external and match existing runtime expectations.

## Deliverables
- Updated package publishing contract in `packages/core/package.json`.
- New core library build config and adjusted root build tooling.
- Internal source diagnostic fixes.
- Updated docs + changelog.
- Verified build/check/test/pack outputs confirming downstream-safe consumption.
